# /meeting - 团队会议命令

## 命令说明

`/meeting` 用于召开内部团队会议，所有角色使用中文进行专业讨论，充分发挥各自专长，最终由PM做出综合决策。

**用途**: 关键决策讨论、方案评审、问题解决、经验分享
**特点**: 中文交流、专业分析、工具辅助、民主决策

## 参数说明

```bash
/meeting "会议主题"
```

**参数**: 会议要讨论的主题或议题

**示例**:
```bash
/meeting "技术栈选择讨论"
/meeting "架构方案评审"
/meeting "项目复盘总结"
/meeting "需求优先级排序"
```

## 会议类型

### 1. 技术方案评审会

```markdown
**主题**: "技术方案评审"
**重点角色**: AR (主讲), PDM, LD
**关注点**:
- 架构设计合理性
- 技术选型正确性
- 实现可行性
- 性能和安全考虑
```

### 2. 需求讨论会

```markdown
**主题**: "需求讨论与澄清"
**重点角色**: PDM (主讲), AR, PM
**关注点**:
- 需求清晰度
- 用户价值
- 优先级排序
- 技术可行性
```

### 3. 项目规划会

```markdown
**主题**: "项目规划与任务分解"
**重点角色**: PM (主持), PDM, AR, LD
**关注点**:
- 任务分解合理性
- 依赖关系
- 时间估算
- 风险识别
```

### 4. 问题解决会

```markdown
**主题**: "技术问题讨论"
**重点角色**: AR, LD, 全员参与
**关注点**:
- 问题根本原因
- 解决方案选项
- 方案评估
- 实施计划
```

### 5. 项目复盘会

```markdown
**主题**: "项目复盘总结"
**重点角色**: PM (主持), 全员参与
**关注点**:
- 项目成果回顾
- 成功经验总结
- 问题和教训
- 改进建议
```

## MCP工具使用

### 会议中的工具协作

```markdown
**PDM使用**:
- context7: 需求上下文分析
- mcp-deepwiki: 行业最佳实践查询

**AR使用**:
- sequential-thinking: 技术方案系统分析
- mcp-deepwiki: 技术文档研究

**DW使用**:
- memory: 历史经验查询
  memory.recall({query: "[会议主题]", limit: 5})

**PM使用**:
- mcp-server-time: 记录会议时间
- shrimp-task-manager: 记录行动项
```

### 会议后的跟进

```markdown
**PM**:
1. 使用shrimp-task-manager创建行动项任务
2. 分配责任人和截止日期
3. 设置跟进提醒

**DW**:
1. 创建会议记录: /project_document/meetings/[主题]_[日期].md
2. 使用模板: /.claude/templates/meeting-minutes-template.md
3. 使用mcp-git-ops提交文档
```

## 使用示例

### 示例1: 技术栈选择

```bash
用户: /meeting "技术栈选择讨论"

PM: 说明背景 - 为新项目选择技术栈

PDM: [context7分析] 需求特点、用户场景
AR: [sequential-thinking] 系统分析三种技术栈的优劣
LD: 评估实现难度和开发效率
DW: [memory] 回忆历史项目经验
QE: 评估测试难度和质量风险

PM: 综合决策 - 选择React + Node.js + PostgreSQL
    理由: [综合各方意见]
    行动: AR完成详细技术方案

输出: 会议记录 + 技术栈决策文档
```

### 示例2: 架构方案评审

```bash
用户: /meeting "架构方案评审"

PM: 说明 - AR已完成架构设计，请大家评审

AR: [sequential-thinking] 详细讲解架构设计
    - 总体架构
    - 模块划分
    - 技术选型
    - 风险预案

PDM: 从产品角度评估方案是否满足需求
LD: 从实现角度评估可行性
QE: 从质量角度评估测试策略
DW: [memory] 类似架构的历史经验

PM: 决策 - 方案通过，AR补充安全设计
    行动: AR更新架构文档

输出: 会议记录 + 评审意见 + 改进要求
```

### 示例3: 项目复盘

```bash
用户: /meeting "用户管理系统项目复盘"

PM: 回顾项目目标、过程、成果

PDM: 需求实现情况、用户反馈
AR: 技术方案回顾、架构经验
LD: 实现过程总结、技术难点
DW: [memory] 记录经验教训
QE: 质量情况总结

PM: 综合总结
    - 成功经验
    - 需要改进的地方
    - 给未来项目的建议

DW: 使用memory提交项目经验

输出: 复盘报告 + 经验沉淀
```

## 注意事项

1. **中文交流**
   - 所有角色都使用中文进行讨论
   - 保持专业性和清晰度

2. **工具使用标注**
   - 明确标注使用的MCP工具
   - 格式: [INTERNAL_ACTION: Action via tool.]

3. **角色发言顺序**
   - 通常: PM开场 → PDM → AR → LD → DW → QE → PM决策
   - 可根据会议主题调整顺序

4. **决策民主性**
   - 充分听取各角色意见
   - PM综合决策并说明理由

5. **行动可追踪**
   - 明确责任人和截止日期
   - 使用shrimp-task-manager跟踪

6. **会议记录**
   - DW负责整理会议记录
   - 存档到project_document/meetings/

## 关键原则

**专业分工** - 各角色发挥专长
**工具赋能** - 使用MCP工具辅助分析
**民主决策** - 充分讨论，综合决策
**行动导向** - 明确行动项和责任人
**知识沉淀** - 记录会议内容
